Database Normalization

Organizing database tables to reduce redundancy and improve integrity

📖

Topic Explanation

Understand the concept clearly

Database Normalization is the process of structuring a relational database to minimize data redundancy and avoid data anomalies (insertion, update, deletion). It involves decomposing large tables into smaller, well-structured tables linked by relationships.

Normalization follows a series of Normal Forms (NF), each building on the previous one:

Normal FormRuleExample Fix
1NFAtomic values; no repeating groups; each column holds single valuesSplit "Phone1, Phone2" into separate rows
2NF1NF + No partial dependencies (all non-key columns depend on the full primary key)Move student name out of an order table
3NF2NF + No transitive dependencies (non-key columns depend only on the PK)Move city info to a separate ZIP table
BCNFStricter 3NF — every determinant must be a candidate keyResolve instructor-course conflicts
📋 1NF Atomic columns
📋 2NF No partial deps
📋 3NF No transitive deps
📋 BCNF Candidate key rule
📝

Summary Notes

Quick revision bullet points

  • Normalization reduces data redundancy and prevents anomalies (insert/update/delete issues).
  • 1NF: Each column must have atomic (indivisible) values; no repeating groups allowed.
  • 2NF: Must be in 1NF; every non-key attribute must depend on the entire primary key (remove partial dependencies).
  • 3NF: Must be in 2NF; no non-key attribute should depend on another non-key attribute (remove transitive dependencies).
  • BCNF: A stronger version of 3NF where every functional determinant is a candidate key.
  • Denormalization is sometimes applied intentionally to improve read performance at the cost of redundancy.

Primary Key vs Foreign Key

Understanding relational integrity constraints

📖

Topic Explanation

Understand the concept clearly

A Primary Key (PK) uniquely identifies each record in a table. No two rows can have the same PK value and it cannot be NULL. A Foreign Key (FK) is a column (or set of columns) in one table that references the Primary Key of another table, establishing a link between the two tables.

FeaturePrimary KeyForeign Key
PurposeUniquely identifies a rowLinks two tables together
UniquenessMust be uniqueCan have duplicates
NULL valuesNot allowedAllowed (optional)
Count per tableOnly one PK per tableCan have multiple FKs
ReferencesSelf-containedReferences PK of another table
-- Primary Key
CREATE TABLE Students (
  StudentID INT PRIMARY KEY,
  Name VARCHAR(100)
);

-- Foreign Key
CREATE TABLE Enrollments (
  EnrollID INT PRIMARY KEY,
  StudentID INT FOREIGN KEY REFERENCES Students(StudentID)
);
📝

Summary Notes

Quick revision bullet points

  • Primary Key: Uniquely identifies each row in a table; cannot be NULL; only one per table.
  • Foreign Key: A field referencing the PK of another table; enforces referential integrity.
  • A table can have only one Primary Key but multiple Foreign Keys.
  • Foreign Keys prevent orphaned records — you can't insert a FK value that doesn't exist in the parent table.
  • Referential Integrity: The constraint that ensures FK values always match a valid PK or are NULL.
  • ON DELETE CASCADE automatically removes child records when the parent record is deleted.

SQL Joins

Combining data from multiple tables using different join types

📖

Topic Explanation

Understand the concept clearly

SQL Joins combine rows from two or more tables based on a related column. They are fundamental for querying relational databases.

Join TypeReturnsUnmatched Rows
INNER JOINRows matching in BOTH tablesExcluded from both sides
LEFT JOINAll rows from left table + matches from rightRight side shows NULL
RIGHT JOINAll rows from right table + matches from leftLeft side shows NULL
FULL OUTER JOINAll rows from both tablesNULLs on both sides
CROSS JOINCartesian product of both tablesNo condition needed
SELF JOINA table joined with itselfUsed for hierarchical data
-- INNER JOIN Example
SELECT s.Name, e.Course
FROM Students s
INNER JOIN Enrollments e ON s.StudentID = e.StudentID;

-- LEFT JOIN Example
SELECT s.Name, e.Course
FROM Students s
LEFT JOIN Enrollments e ON s.StudentID = e.StudentID;
📝

Summary Notes

Quick revision bullet points

  • INNER JOIN: Returns only the rows where there is a match in both tables.
  • LEFT JOIN: Returns all rows from the left table, with NULLs for non-matching right rows.
  • RIGHT JOIN: Returns all rows from the right table, with NULLs for non-matching left rows.
  • FULL OUTER JOIN: Returns all rows from both tables; NULLs fill in where no match exists.
  • CROSS JOIN: Returns the Cartesian product — every combination of rows from both tables.
  • Joins are performed using the ON clause to specify the matching condition.

ACID Properties

The four pillars of reliable database transactions

📖

Topic Explanation

Understand the concept clearly

ACID is a set of properties that guarantee database transactions are processed reliably. These properties are critical for systems where data integrity is essential (e.g., banking, e-commerce).

PropertyMeaningExample
AtomicityAll operations in a transaction succeed or all fail — no partial executionBank transfer: debit + credit both succeed or both rollback
ConsistencyA transaction brings the database from one valid state to anotherAccount balance never goes negative if a rule prevents it
IsolationConcurrent transactions don't interfere with each otherTwo users booking the last seat don't both succeed
DurabilityOnce committed, a transaction is permanently saved even after system failurePayment confirmed survives a server crash
🔵 Atomicity "All or nothing"
🟢 Consistency "Valid state always"
🟡 Isolation "Independent"
🔴 Durability "Permanent commit"
📝

Summary Notes

Quick revision bullet points

  • Atomicity: A transaction is treated as a single unit — it either fully completes or fully rolls back.
  • Consistency: Every transaction maintains the database integrity rules and constraints.
  • Isolation: Concurrent transactions are executed as if they were sequential; results are hidden until committed.
  • Durability: Committed data persists permanently, even in case of power failure or crash.
  • ACID is ensured by DBMS features like transaction logs, rollback mechanisms, and locks.
  • NoSQL databases sometimes sacrifice ACID for performance, offering BASE (Basically Available, Soft-state, Eventually consistent).

Data Structures

Organizing and storing data efficiently for optimal access

📖

Topic Explanation

Understand the concept clearly

A Data Structure is a way of organizing, storing, and managing data in memory so it can be accessed and modified efficiently. Choosing the right data structure is critical for algorithm performance.

StructureAccessInsertBest Use Case
ArrayO(1)O(n)Fixed-size sequential data
Linked ListO(n)O(1)Frequent insertions/deletions
StackO(1) topO(1)Undo operations, backtracking
QueueO(1) frontO(1)Scheduling, BFS traversal

Binary TreeO(log n)O(log n)Searching and sorting
GraphO(V+E)O(1)Networks, maps, social connections
📝

Summary Notes

Quick revision bullet points

  • Data structures are classified as Linear (Array, Stack, Queue, Linked List) and Non-Linear (Trees, Graphs).
  • Arrays offer O(1) random access but slow insertion/deletion at arbitrary positions.
  • Hash Tables provide average O(1) access and insertion using a hash function and collision handling.
  • Trees represent hierarchical data; Binary Search Trees allow O(log n) search.
  • Graphs model networks and relationships; traversed with BFS or DFS algorithms.
  • Choosing the correct data structure directly impacts the time and space complexity of a program.

Stack vs Queue

Two fundamental linear data structures with opposite ordering rules

📖

Topic Explanation

Understand the concept clearly

A Stack follows LIFO (Last In, First Out) — the last element added is the first to be removed. Think of a stack of plates. A Queue follows FIFO (First In, First Out) — the first element added is the first removed. Think of a line at a ticket counter.

FeatureStackQueue
OrderLIFO (Last In First Out)FIFO (First In First Out)
Insert operationPush (top)Enqueue (rear)
Remove operationPop (top)Dequeue (front)
PeekTop elementFront element
Real-world useUndo, Recursion call stack, Browser backPrint queue, BFS, CPU scheduling
-- Stack (Python)
stack = []
stack.append(1) # Push
stack.pop() # Pop → returns 1

-- Queue (Python)
from collections import deque
queue = deque()
queue.append(1) # Enqueue
queue.popleft() # Dequeue → returns 1
📝

Summary Notes

Quick revision bullet points

  • Stack = LIFO: Last In, First Out. Operations: Push (add), Pop (remove), Peek (view top).
  • Queue = FIFO: First In, First Out. Operations: Enqueue (add), Dequeue (remove), Front (view first).
  • Stack use cases: function call stack, undo/redo, expression evaluation, browser history.
  • Queue use cases: CPU task scheduling, print spooling, BFS graph traversal, message queues.
  • Circular Queue: A queue where the last position connects back to the first — efficient memory usage.
  • Priority Queue: A queue where each element has a priority; higher priority elements are dequeued first.
✅

Linked List

A dynamic linear data structure made of connected nodes

📖

Topic Explanation

Understand the concept clearly

A Linked List is a linear data structure where each element (called a node) contains data and a pointer to the next node. Unlike arrays, linked lists don't require contiguous memory allocation, making insertions and deletions efficient.

TypeStructureTraversal
Singly LinkedEach node → next node onlyForward only
Doubly LinkedEach node → next + previousForward and backward
Circular LinkedLast node points back to headLoops continuously
// Node Structure (C-style)
struct Node {
  int data;
  struct Node* next;
};

// Singly Linked List: Head → [1|→] → [2|→] → [3|NULL]
➕ Insert O(1) At head or tail
🔍 Search O(n) Sequential traversal
❌ Delete O(1) With pointer access
💾 Dynamic Size No fixed allocation
📝

Summary Notes

Quick revision bullet points

  • A Linked List is a chain of nodes; each node stores data and a reference (pointer) to the next node.
  • Singly Linked List: One direction traversal; each node has next pointer only.
  • Doubly Linked List: Bi-directional traversal; each node has both next and previous pointers.
  • Circular Linked List: The last node's next pointer points back to the head node.
  • Advantage over arrays: Dynamic size; O(1) insertion/deletion at known positions.
  • Disadvantage: No random access — searching requires O(n) traversal from the head.
✅

Binary Trees

Hierarchical structure where each node has at most two children

📖

Topic Explanation

Understand the concept clearly

A Binary Tree is a tree data structure where each node has at most two children: a left child and a right child. A Binary Search Tree (BST) adds the rule: left child values are smaller than the parent, and right child values are greater.

TraversalOrderOutput (example tree)
InorderLeft → Root → RightSorted ascending (BST)
PreorderRoot → Left → RightUsed to copy a tree
PostorderLeft → Right → RootUsed to delete a tree
Level OrderLevel by level (BFS)Breadth-first traversal
// BST structure
      8
    /   \
   3    10
  / \    \
 1  6    14
// Inorder: 1, 3, 6, 8, 10, 14 (sorted!)
📝

Summary Notes

Quick revision bullet points

  • A Binary Tree has at most 2 children per node (left and right).
  • BST Rule: Left subtree values < root; right subtree values > root.
  • BST Search, Insert, Delete: O(log n) on average; O(n) worst case (unbalanced).
  • Inorder traversal of a BST gives values in sorted (ascending) order.
  • Balanced BSTs (AVL Tree, Red-Black Tree) maintain O(log n) height for guaranteed performance.
  • Binary Trees are used in expression parsing, Huffman encoding, and database indexing (B-Trees).

Time Complexity (Big-O)

Measuring algorithm efficiency as input size grows

📖

Topic Explanation

Understand the concept clearly

Big-O Notation describes the worst-case time or space an algorithm uses as input size (n) grows. It helps compare algorithms independently of hardware or language differences.

NotationNameExampleSpeed
O(1)ConstantArray index access⚡ Fastest
O(log n)LogarithmicBinary search🚀 Very fast
O(n)LinearLinear search✅ Good
O(n log n)LinearithmicMerge sort, Quick sort🟡 Acceptable
O(n²)QuadraticBubble sort, nested loops🔴 Slow
O(2ⁿ)ExponentialRecursive Fibonacci💀 Very slow
O(n!)FactorialTravelling Salesman (brute force)❌ Avoid
⚡ O(1) Best — Constant
🚀 O(log n) Binary Search
✅ O(n) Linear scan
🔴 O(n²) Nested loops
📝

Summary Notes

Quick revision bullet points

  • Big-O describes the worst-case performance of an algorithm as n (input size) approaches infinity.
  • O(1): Constant time — doesn't depend on input size (e.g., accessing an array element by index).
  • O(log n): Logarithmic — input is halved each step (e.g., Binary Search on sorted array).
  • O(n): Linear — time grows proportionally with input (e.g., looping through a list once).
  • O(n²): Quadratic — common with nested loops; doubles in cost exponentially (e.g., Bubble Sort).
  • Space Complexity: Also expressed in Big-O; measures memory usage relative to input size.
  • Always aim to reduce complexity: O(n²) → O(n log n) → O(n) → O(log n) → O(1).
✅
